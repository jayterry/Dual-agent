台灣可信度評分（逐點說明版）
以下是我們整理出的完整配分邏輯，已針對台灣情境做調整，包含中文語料的偵測覆蓋與每一項配分的原因。


TIER 1（最難偽造；可作為決定性依據）
A）惡意 URL 證據 — 0 到 40 分
判定條件：URL 經威脅情資（Threat Intelligence）確認為惡意（全球資料源 + 若可用則含台灣在地資料源）。
配分方式（依證據強度）：
未被任何來源標記：0 分
被 1 個來源標記：15 分
被 2–3 個來源標記：25 分
被 4+ 個來源標記，或多家廠商高度一致：35 分
被 5+ 個來源標記，且明確標註為釣魚/惡意軟體：40 分
為何可信：屬於外部可驗證、客觀的證據，誤報率極低。
台灣注意事項：
台灣常見詐騙會用短網址 → 務必先展開（expand）再檢查
常見仿冒 .com.tw 品牌的相似網域（lookalike domain）


B）要求密碼 / 網銀密碼 — 0 或 25 分
判定條件：訊息要求提供密碼或帳號憑證（account credentials）。
需支援的繁中表達（例）：
「驗證/確認/更新/變更 密碼」
「請輸入/請提供 密碼/帳密/登入密碼」
「網路銀行 密碼/帳密 驗證」
為何一旦出現就給 25 分：這是幾乎通用的「合法服務不會透過簡訊索要密碼」規則，誤報率極低。


C）要求卡片 / 銀行敏感資料 — 0 或 25 分
判定條件（包含但不限於）：
信用卡/金融卡卡號
CVV/安全碼
到期日
銀行帳號/匯款資料
繁中表達（例）：
「信用卡/金融卡/銀聯卡 + 卡號/安全碼/驗證」
「請提供 卡號/CVV」
為何 25 分：透過簡訊要求這類資料幾乎都是詐騙。


D）要求身分證字號 / 報稅身分識別 — 0 或 25 分
（台灣情境取代 “SSN”）
判定條件（包含但不限於）：
「身份證/身分證 字號/號碼/編號」
“National ID number”
以「補助/退稅/銀行驗證」等名義要求提供身分證字號
為何 25 分：在簡訊中要求身分證字號是高度可信的詐騙訊號（通常指向身分盜用或詐財）。


E）要求本地支付 App 的 PIN/密碼（LINE Pay / 街口 / 全支付 / 悠遊付 / 一卡通Money / 台灣Pay）— 0 或 25 分
判定條件：任何要求提供或驗證 PIN/密碼/交易密碼的訊息。
為何 25 分：這些支付 App 不應在未經使用者主動操作的情況下，透過簡訊索要 PIN/密碼。


F）SSL 憑證無效 / 網域不相符 — 0 到 15 分
前提：訊息包含 URL，且符合下列情況之一：
自簽憑證（self-signed）：15 分
憑證網域不相符（domain mismatch）：15 分
憑證過期（expired certificate）：10 分
非 HTTPS（no HTTPS）：5 分
台灣注意事項：許多詐騙站使用廉價/免費網域，TLS 常見設定不良或異常。


TIER 2（非常可信；用於加強判斷）
G）發件者冒充（技術驗證）— 0 到 20 分
判定條件：透過網路/電信層面可觀測的資訊，發現發件者身份與宣稱不符；或來源路由可疑（VoIP、海外）但內容宣稱為台灣本地銀行/政府/物流等。
配分範例：
VoIP 來源 + 宣稱「銀行/政府/物流」：15 分
海外來源不合理（宣稱本地台灣機構但來源在海外）：12 分
發件者不在已驗證的商業 Sender ID 白名單（若可取得）：10 分
已驗證的商業簡訊代碼且與品牌匹配：0 分
為何不是 Tier 1：部分合法機構可能使用第三方簡訊服務或特殊路由，因此不一定 100% 決定性。


H）台灣在地詐騙框架匹配（高信心）— 0 到 20 分
我們定義的台灣常見詐騙框架包含：
政府補助/補償/津貼
物流/包裹/派送失敗
銀行帳戶異常/凍結
eTag/水費/電費/瓦斯費/欠費
投資/加密貨幣/保證獲利/群組
OTP/驗證碼詐騙（特別危險）
依匹配信心度配分：
95%+ 匹配：20 分
85–95%：15 分
75–85%：10 分
50–75%：5 分
<50%：0 分
避免亂評分的關鍵護欄：只有在信心度高時才給高分。


I）台灣語境下的文法/拼字錯誤群聚 — 0 到 12 分
台灣語言品質指標（例）：
繁簡混用（在台灣簡訊中出現大量簡體字通常可疑）
大量奇怪符號/標點
多處錯字（3+）顯示低品質詐騙模板
配分規則：
0–2 個問題：0 分
3–4 個問題：8 分
5+ 個問題：10–12 分
若繁簡混用特別明顯：可推到 12 分
為何是 Tier 2 而非 Tier 1：合法訊息也可能有少量錯字；只有「群聚」才有意義。


J）網域年齡 + 台灣 TLD（頂級網域）邏輯 — 0 到 12 分
規則：新網域加分（越新越可疑）
<3 天：12 分
3–7 天：8 分
7–30 天：5 分
30 天以上：0 分
可疑/便宜/常被濫用的 TLD
.tk .ml .ga .cf .top .info → 通常較可疑
台灣常見/較正規的網域型態
.com.tw, .tw, .org.tw, .gov.tw
（.gov.tw 通常高度可信，但仍建議驗證）


TIER 3（中等可信；不得單獨使用）
K）緊急/威脅語氣（繁中）— 0 到 5 分（僅在 Tier 1 或 Tier 2 已觸發時可加）
繁中緊急關鍵字（例）：
「緊急、立即、馬上、現在、截止、期限、倒數、最後期限、否則、將被、停止、凍結」
配分規則：
單獨只有緊急語氣：0 分（避免誤報）
緊急語氣 + 其他可信訊號（Tier 1/2）：最多 5 分


Tier 4（避免作為主要依據）
避免僅依下列特徵給大量分數：
只靠「權威宣稱」（例如僅說「我是銀行」）
單獨情緒用語
單獨要求保密/隔離（secrecy/isolation）
原因：這些特徵誤報率高，容易造成亂評分。






  



Prompt: 輸入（Input）
你會收到一個 JSON：


text: 簡訊/通知全文（可能含繁中/英文/混合）
sender_display: 使用者手機上看到的發件者名稱或號碼（可能是短碼、商業代碼、名稱）
sender_number: 可能的來電號碼（若無則空）
urls: 從 text 萃取出的 URL 清單（若無則空陣列）
expanded_urls: 對短網址展開後的 URL 清單（若無則空陣列）
url_threat_hits: 威脅情資命中結果（由系統提供，不要你猜），格式：
每個 URL: { "url": "...", "hits": { "virustotal": true/false, "phishtank": true/false, "urlhaus": true/false, "taiwan_165": true/false, "telco_blocklist": true/false }, "vendor_count": N, "label": "phishing/malware/clean/unknown" }
tls_findings: TLS/SSL 檢測結果（由系統提供，不要你猜），格式：
每個 URL: { "url": "...", "https": true/false, "self_signed": true/false, "domain_mismatch": true/false, "expired": true/false }
sender_tech_context: 發件者技術資訊（若系統提供），例如：
{ "route_type": "carrier/voip/unknown", "origin_country": "TW/other/unknown", "sender_id_verified": true/false/unknown, "commercial_code_digits": 0/5/6/7/unknown }
核心原則（必須遵守）
只用可驗證/可觀測的證據加分。沒有證據就給 0。
Tier 3（緊急語氣）不可單獨加分：只有當 Tier 1 或 Tier 2 至少命中一項時，Tier 3 才允許加最多 5 分。
Tier 4（權威宣稱/情緒/要求保密）不可作為主要加分依據：最多只能寫在解釋文字中，不得給高分。
分數計算後請 clamp 到 0~100，輸出 r_llm。
你必須輸出「每一項加分的證據片段」：引用原文片段（quote）或引用系統提供的檢測結果欄位（例如 vendor_count、self_signed）。
不要把「可能是詐騙」當作證據；證據必須是「命中規則」。
台灣版 Tier 規則與配分（Point Rules）
TIER 1（最難偽造，可決定性）
A) 惡意 URL 證據（0~40）


若 URL 未被任何來源標記：0
命中 1 個來源：15
命中 2~3 個來源：25
命中 4+ 來源 或 vendor_count 很高：35
命中 5+ 來源 且 label = phishing/malware：40 （優先使用 url_threat_hits，不要自行猜測）
B) 要求密碼/帳密（0 或 25）


若 text 明確要求提供/驗證/更新密碼或帳密：25 否則 0 （繁中關鍵語意例：驗證密碼、確認密碼、請輸入密碼、帳號密碼、網銀密碼、登入密碼）
C) 要求卡號/CVV/到期日/匯款資料（0 或 25）


明確索取信用卡/金融卡敏感資料或匯款資訊：25 否則 0
D) 要求身分證字號/身分識別（0 或 25）


明確索取「身分證字號/身分證號碼/國民身分證」或等價身分識別：25 否則 0
E) 要求本地支付 App PIN/密碼（0 或 25）


明確索取 LINE Pay/街口/全支付/悠遊付/一卡通Money/台灣Pay 的 PIN/交易密碼/帳密：25 否則 0
F) TLS/SSL 異常（0~15，需有 URL）


self_signed = true：15
domain_mismatch = true：15
expired = true：10
https = false：5 （使用 tls_findings，不要自行猜）
TIER 2（非常可信，加強判斷）
G) 發件者冒充（0~20） 根據 sender_tech_context：


route_type=voip 且訊息宣稱銀行/政府/物流：15
origin_country != TW 且訊息宣稱本地台灣機構：12
sender_id_verified=false 或 unknown 且屬明顯品牌宣稱：10
sender_id_verified=true 或合規商業代碼且匹配：0 （若 sender_tech_context 缺失，請不要硬加分）
H) 台灣詐騙框架匹配（0~20） 只在你能提供「清楚的框架特徵」時才加分，並用信心分級：


95%+：20
85~95：15
75~85：10
50~75：5
<50：0 框架類型至少包含：
政府補助/補償/津貼
物流/包裹/派送失敗
銀行帳戶異常/凍結
eTag/水電瓦斯欠費
投資/加密貨幣/保證獲利/群組
OTP/驗證碼詐騙（特別危險） （注意：框架配分不能取代 Tier1 的硬證據；若只靠語氣模糊，請低分或 0）
I) 文法/錯字群聚（0~12）


0~2 個問題：0
3~4：8
5+：10~12
若繁簡混用非常明顯，可到 12 （若沒有可靠的錯誤數據，請只在你能引用具體錯誤時才加分）
J) 網域年齡/可疑 TLD（0~12） 若系統提供 domain age / TLD risk 才可加分；未提供則不加。


<3 天：12
3~7 天：8
7~30：5
30+：0
.tk .ml .ga .cf .top .info 可視作加強可疑（但需證據支持）
TIER 3（中等可信；不得單獨使用）
K) 緊急/威脅語氣（0~5） 只有當 Tier 1 或 Tier 2 至少命中一項，才可加最多 5 分。 常見詞：緊急、立即、馬上、現在、截止、期限、倒數、最後期限、否則、將被、停止、凍結


TIER 4（避免作主要依據）
權威宣稱/情緒語/要求保密：


不得作為主要加分原因
只能寫在 explain/reason 裡作輔助描述
